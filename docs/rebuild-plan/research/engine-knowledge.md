## 8. Needs-you hygiene (standing operational habit)

- **Scar:** the "needs my input" pile builds up and "gets very much in the way"; runs start clogged.
- **RULE (Pierre's standing instruction 2026-06-27):** before/at start of any auto-apply run, clear the needs-you queue: (1) categorize — `captcha`/`login` are NOT answerable → terminal `skipped` (site-blocked); `awaiting_review` are usually already-submitted (v11.61) → `done`; (2) answer real screening Qs from profile+memory (work auth = authorized, no sponsorship; relocation = open; availability = 2 weeks; referral = "No"); save via profile-fields (locks, conf 1, reused forever) or intake (answers→profile→re-queue); (3) NEVER fabricate sensitive/EEO answers or specifics not in the profile; (4) report counts cleared.
- **Design implication:** answered needs-you re-queues at the FRONT of its lane (v13); the bigger the learned memory, the fewer parks — memory growth IS the submit-rate lever (§2.3).

## 11. Run lifecycle: resume-by-reclassification (the MV3 answer)

- **Scar:** v11's terminal failure class — all intelligence in a ~4k-line MV3 content script Chrome constantly kills; "Easy Apply form disappeared after advancing" ×155/4h → 1 apply/2.5h + app freeze. This is WHY v12/v13 exist.
- **RULES (proven in v12 M1 survival tests, carried into v13):**
  - App owns the FSM (13 states, DB columns); extension is stateless hands+eyes (snapshots up, idempotent commands down).
  - SW eviction/port death/bfcache = NORMAL events: port death → `waiting_page` → on reconnect **RE-CLASSIFY the live page and continue from where the page actually IS** (resume_count++). **Never replay** command history.
  - TabSession **epochs** (ulid per tabId×committed-navigation) on every snapshot/event/command; a stale-epoch command is refused (`stale_epoch`) — a command aimed at a dead page can never fire on a new one. bfcache restore = old epoch (resume); real nav = new epoch (re-classify).
  - Watchdog TTLs per state; `waiting_page` 120s (v11.84's stranded-slot lesson made structural); whole-run hard cap 8min EXCEPT `needs_human` (§5.5).
  - Step-graph-aware classification (prefer successor>same>other) resolves ambiguous pages; hot-reloaded adapters take effect on the next lease, in-flight runs keep their pinned version.

## 16. Debugging discipline (meta-rules that saved every diagnosis above)

- **Never debug blindly (Pierre's hard rule, 2026-06-20):** never ship a fix not grounded in DIRECT observation of the real failure. The occlusion saga = 3 wasted releases off coarse transcripts. If blind → STOP and ask for observability (pause the run to drive the real page; verbose debug build; capture the DOM as a fixture). Build observability IN: no terminal state without a diagnostic, trace-level logging (redacted), heap/route telemetry.
- **Observe-first beats synthesis:** v11.57's one-line keyword fix beat a proposed 7-point executor rewrite; build the fixture, run current code against it, then minimal fix.
- **Harness before live:** the Playwright harness (`v11/harness/`) runs the REAL executor against faithful fixtures under real hostnames — verify every apply-path change there before any live run (Pierre's hard rule: verify auto-submission yourself, multiple times, before deploying). Harness can't reproduce OS focus wars, Cloudflare, or pool-side tab reuse — those need unit-proofs by construction + a watched live run.
- **Live-DB forensics:** copy jat.db **+ -wal + -shm** (WAL: db-only copies are STALE) or query the REST API (token in kv.authToken). Measure execution OVERLAP (scheduled_at…updated_at spans), not scheduled_at clustering, when judging concurrency. Check the jobspy discovery-batch TIMELINE (gaps = starvation) when long-run throughput collapses.
- **Perf playbook (v11.82):** find the process by port owner → sample idle CPU (idle ≈ event-driven) → `curl -w` time AND size every endpoint → per-column byte probe for blob hogs → EXPLAIN QUERY PLAN (include sqlite_autoindex) → re-measure end-to-end from source on an alt port before shipping.
